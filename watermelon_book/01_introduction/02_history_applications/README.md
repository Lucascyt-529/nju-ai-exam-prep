# 绪论2：发展脉络与应用判断

这两节偏知识框架，不适合为了“有代码”而制造一个伪算法。本专题用时间线、范式比较和任务拆解覆盖教材第1.5～1.6节，并用迁移问题检查是否真正理解。

## 一、发展脉络不是算法名单

可以把机器学习的发展看成几个反复交织的问题：

1. **推理能力够不够？** 早期人工智能重视逻辑推理和问题求解，但很快发现“能推理”不等于“拥有解决现实问题所需的知识”。
2. **知识从哪里来？** 知识工程和专家系统把人的规则写进机器，却暴露出知识获取、维护和迁移成本高的问题；让机器从数据中获得规律因此变得重要。
3. **应该学规则、连接还是概率关系？** 符号主义强调可解释的规则与概念，连接主义强调由网络连接和权重形成表示，统计学习强调从样本与不确定性出发控制泛化。三者并非简单地互相消灭。
4. **为什么同一类思想会再次兴起？** 数据规模、计算能力、存储条件、优化方法和任务需求变化，会让旧思想在新条件下展现不同效果。神经网络的多次兴起是典型例子。

### 一个便于复述的时间框架

| 阶段 | 主要关注 | 典型瓶颈 | 对机器学习的推动 |
| --- | --- | --- | --- |
| 20世纪50年代至70年代初 | 逻辑推理、通用问题求解 | 缺少现实领域知识 | 意识到系统需要获得和利用知识 |
| 70年代中期以后 | 知识工程、专家系统 | 规则获取与维护困难 | 学习机制从辅助角色变成核心问题 |
| 80年代 | 机器学习形成更独立的研究领域 | 不同范式各有适用边界 | 符号、连接等学习路线系统发展 |
| 90年代以后 | 统计学习、泛化与核方法 | 模型假设和真实任务是否匹配 | 概率统计与优化成为重要基础 |
| 21世纪初以来 | 大数据、深层网络与工程规模化 | 数据/算力成本、可靠性与解释 | 表示学习和复杂模型广泛应用 |

这张表是理解索引，不是要求背诵年份。考试更可能问“某次范式变化解决了什么瓶颈”，而不是孤立地默写人物列表。

## 二、如何分析一个机器学习应用

不要只说“这个场景可以用机器学习”。至少按下面六个问题拆解：

1. **样本是什么？** 一个人、一张图、一次交易还是一段时间窗？
2. **特征从哪里来？** 原始传感器、人工字段、文本表示还是历史统计？
3. **监督信号是什么？** 类别、连续值、成对偏好，还是没有标签？
4. **输出是什么任务？** 分类、回归、排序、聚类、降维或异常检测？
5. **评价代价是什么？** 错误是否对称，是否更怕漏报，是否要求概率校准？
6. **部署约束是什么？** 延迟、内存、隐私、分布漂移、可解释性与人工复核。

### 应用映射示例

| 场景 | 样本/输入 | 任务 | 不能忽略的约束 |
| --- | --- | --- | --- |
| 垃圾邮件识别 | 一封邮件的文本与元数据 | 二分类 | 误拦正常邮件与漏掉垃圾邮件代价不同 |
| 房价估计 | 一套房屋的属性 | 回归 | 时间和地区分布漂移，异常成交价 |
| 客户分群 | 用户行为统计 | 聚类 | 簇没有天然“正确标签”，需解释用途 |
| 搜索排序 | 查询—文档对 | 排序 | 点击标签有位置偏差，线上延迟严格 |
| 医学辅助筛查 | 检查数据与病史 | 分类/风险评分 | 高风险决策、校准、隐私和人工复核 |
| 设备预警 | 传感器时间窗 | 异常检测/分类 | 极端类别不平衡、提前量与误报成本 |

## 三、机器学习不等于整个系统

现实任务通常是：

```text
问题定义 -> 数据采集 -> 清洗与划分 -> 建模 -> 评估 -> 部署 -> 监控与更新
```

模型只是中间一环。数据泄漏、错误评价指标或部署分布变化，都可能让离线分数很高的模型失效。这也是本仓库为什么把文件读写、数据处理和无OJ自检放在与算法同等重要的位置。

## 四、学习方式

先独立完成同目录 `exercises.md`，再对照 `reference/answers.md`。参考答案给的是判断路径，不是唯一措辞；能换一个新场景仍按六个问题拆解，才算真正会用。
