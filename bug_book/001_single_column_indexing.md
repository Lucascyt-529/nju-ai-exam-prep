# 001：单列索引后为什么不是二维列向量

## 发生场景

第一次 NumPy 现场诊断中，对矩阵：

```python
X = np.array([
    [1.0, 10.0, 100.0],
    [2.0, 20.0, 200.0],
    [3.0, 30.0, 300.0],
])
```

学习者判断 `X[:, 1]` 的值为 `[[1.0], [2.0], [3.0]]`，形状为 `(3, 1)`。

## 具体问题

这里混淆了两个概念：

1. 下标 `1` 选择的是第二列，而不是第一列；
2. 使用整数下标选择单列会消去这一维，结果是一维数组。

实际结果：

```python
X[:, 1]       # array([10., 20., 30.])
X[:, 1].shape # (3,)
```

如果需要保留二维列向量，应使用切片或列索引列表：

```python
X[:, 1:2]        # shape == (3, 1)
X[:, [1]]        # shape == (3, 1)
```

## 根本原因

“从表格中取一列”是语义描述，不代表 NumPy 一定保留二维形状。整数索引会降维，切片通常保留被切片的轴。

## 最小反例

```python
X = np.array([[1, 10], [2, 20], [3, 30]])

assert X[:, 1].shape == (3,)
assert X[:, 1:2].shape == (3, 1)
```

两者元素数量相同，但广播和矩阵运算行为可能完全不同。

## 今后如何检查

1. 看到单列选择后立即写出 `.shape`；
2. 明确后续函数需要 `(n,)` 还是 `(n, 1)`；
3. 不为“让程序运行”盲目 reshape，要先说明目标形状；
4. 在梯度、标签和预测值运算前并排检查形状。

## 当前状态

已完成迁移验证。学习者能够正确判断：

```python
X[:, 1] - y    # shape == (3,)
X[:, 1:2] - y  # shape == (3, 3)
```

并能说明第二种写法通常违背“预测值与标签逐样本对应”的计算意图。随后已补充理解：二维列数组 `(3, 1)` 与一维数组 `(3,)` 广播时，后者相当于在末尾维度上与前者对齐，因此形成两两相减的 `(3, 3)` 结果。

本条错误已完成第一次解释复盘；仍需在后续损失函数或梯度练习中再次正确处理，才能视为稳定掌握。
